# 工时评估规则

> 基准：1天 = 7.5小时。所有工时以**小时**为单位输出，不换算人天。
> 拆分方式：按功能模块拆分任务，每个模块的工时为综合值（含页面/组件/接口/状态管理），联调单独计算。

---

## 一、功能模块综合工时区间

> 实际输出时不再按维度拆任务，而是直接给出模块综合工时。以下区间供快速参考，推导时可查阅后面各维度的基准表。

| 模块类型 | 典型工作内容 | 综合工时区间 |
|---------|------------|------------|
| CRUD 列表模块 | 列表页 + 筛选 + 分页 + 新增/编辑弹窗 + 接口对接 + Store | 8 ~ 16h |
| 详情展示模块 | 详情页 + 数据展示组件 + 接口对接 | 4 ~ 8h |
| 表单模块 | 表单页 + 校验 + 提交 + 接口对接 | 6 ~ 12h |
| 复杂表单模块 | 多步骤表单 + 联动 + 动态字段 + 接口对接 + Store | 12 ~ 20h |
| 纯展示/配置模块 | 配置页 + 数据渲染 + 接口对接 | 3 ~ 6h |
| 数据导出模块 | 导出按钮 + 文件下载接口 + 进度提示 | 2 ~ 4h |

---

## 二、页面 / 路由维度

> 以下各维度基准表用于推导模块工时时的内部参考，不需要在输出中逐项列出。

| 场景                               | 基准工时 |
| ---------------------------------- | -------- |
| 全新页面（含路由、布局、基础交互） | 4 ~ 8h   |
| 在已有页面基础上新增 Tab / 子页面  | 2 ~ 4h   |
| 纯展示页（无复杂交互，仅渲染数据） | 2 ~ 3h   |
| 表单页（含校验、提交、回显）       | 4 ~ 8h   |
| 复杂表单（多步骤、动态字段、联动） | 8 ~ 15h  |

---

## 三、组件维度

| 组件类型                                 | 基准工时 |
| ---------------------------------------- | -------- |
| 复用已有组件（仅调整 props / 样式）      | 0.5 ~ 1h |
| 基于 UI 库封装业务组件（中等复杂度）     | 2 ~ 4h   |
| 全新业务组件（含完整交互逻辑）           | 4 ~ 8h   |
| 复杂组件（虚拟滚动、拖拽、富文本、图表） | 8 ~ 20h  |
| 弹窗 / 抽屉（含表单或详情）              | 2 ~ 5h   |
| 纯展示卡片 / 列表项                      | 1 ~ 2h   |

---

## 四、接口对接维度

| 场景                                        | 基准工时 |
| ------------------------------------------- | -------- |
| 单个简单接口（GET 列表/详情）               | 0.5 ~ 1h |
| 单个写接口（POST/PUT/DELETE，含提示、刷新） | 1 ~ 2h   |
| 分页 + 筛选接口（含 URL 参数同步）          | 2 ~ 3h   |
| 文件上传 / 下载接口                         | 2 ~ 4h   |
| 复杂接口（轮询、WebSocket、流式响应）       | 4 ~ 8h   |
| 多接口串联（依赖关系、顺序调用）            | 2 ~ 5h   |

---

## 五、状态管理维度

| 场景                                             | 基准工时                   |
| ------------------------------------------------ | -------------------------- |
| 本地组件状态（useState / useRef）                | 不单独计，已含在组件工时内 |
| 新增 Store 模块（Pinia / Zustand / Redux slice） | 2 ~ 4h                     |
| 扩展已有 Store（新增 state / action）            | 0.5 ~ 2h                   |
| 跨页面共享状态 + 持久化                          | 3 ~ 6h                     |
| 复杂状态机（多步骤流程、状态流转）               | 4 ~ 10h                    |

---

## 六、前后端联调维度

联调作为独立任务输出，不拆分到具体接口，按整体接口规模估算：

| 接口数量 | 接口复杂度                         | 参考工时                       |
| -------- | ---------------------------------- | ------------------------------ |
| 1 ~ 3 个 | 简单（增删改查）                   | 2 ~ 4h                         |
| 1 ~ 3 个 | 复杂（文件上传、轮询、多接口串联） | 4 ~ 6h                         |
| 4 ~ 8 个 | 简单为主                           | 4 ~ 8h                         |
| 4 ~ 8 个 | 复杂为主                           | 6 ~ 12h                        |
| 8 个以上 | —                                  | 按比例上浮，考虑拆多个联调任务 |

---

## 七、调整系数

以下情况在基准工时上做乘法调整：

| 情况                                                                   | 系数       |
| ---------------------------------------------------------------------- | ---------- |
| 需求文档清晰，有详细原型                                               | ×1.0       |
| 需求文档模糊，需要自行判断交互细节                                     | ×1.2 ~ 1.5 |
| 需要兼容移动端 + PC 端                                                 | ×1.3       |
| 需要适配国际化（i18n）                                                 | ×1.1 ~ 1.2 |
| 涉及权限控制（按钮级/页面级）                                          | +1 ~ 2h    |
| 新项目 / 不熟悉的代码库（参考 `references/domains/<domain>-reference/SKILL.md` 后仍不清晰） | ×1.3       |
| 有现成可参考的类似页面（`references/domains/<domain>-reference/SKILL.md` 中存在相似实现）   | ×0.7 ~ 0.8 |

---

## 八、评估原则

1. **按功能模块拆分**：以功能模块为任务单位，工时为综合值（包含页面/组件/接口/状态管理），联调单独计算。
2. **结合源码**：若能匹配 domain, 则读取对应 `references/domains/<domain>-reference/SKILL.md` 后，优先参考项目中已有的相似组件/页面，若存在可复用代码则下调工时。
3. **拆小不拆大**：单个任务工时超过 8h，应考虑是否可以进一步拆分。
4. **说明依据**：每个任务附上工时估算的简短理由（1句话），便于评审时讨论。
5. **悲观优于乐观**：不确定时取区间上限，避免低估。
